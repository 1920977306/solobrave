## 对话3：Mock 数据 + 身份注入问题
**链接**: https://www.coze.cn/s/KjK3bZodgrk/

### 问题1：Mock 数据兜底
**症状**: 员工账号没有创建 AI，办公室页面显示假数据（mock 数据）

**根因**: `office-v3.html` 在 API 返回空时走了 mock 数据兜底

**解决方案**: 把 mock 兜底改成空状态提示
```javascript
// 旧代码（错误）
if(data.length === 0){
  // 使用 mock 数据
  data = getMockEmployees();
}

// 新代码（正确）
if(data.length === 0){
  // 显示空状态
  container.innerHTML = '<div class="empty-state">还没有 AI 员工，去主页创建吧</div>';
  return;
}
```

### 问题2：AI 身份注入错误
**症状**: 木木回答自己是"小可"

**根因**: systemPrompt 没有正确注入身份声明（IDENTITY.md 内容）

**解决方案**: 在发送消息前，把 IDENTITY.md 内容注入 system prompt
```javascript
// 在 sendMsg 或 sendViaAPI 中
var identityDoc = emp.docs && emp.docs.IDENTITY ? emp.docs.IDENTITY : '';
var systemPrompt = identityDoc + '\n\n' + (emp.systemPrompt || '');
```

### 问题3：Python 脚本批量修复
**Trumind 用 Python 脚本替换代码**：
```python
import re

with open('index.html', 'r', encoding='utf-8') as f:
    content = f.read()

# 替换 mock 兜底
old = 'if(data.length===0){data=getMockEmployees();}'
new = 'if(data.length===0){container.innerHTML=\'<div class="empty-state">...</div>\';return;}'
content = content.replace(old, new)

with open('index.html', 'w', encoding='utf-8') as f:
    f.write(content)
```

**学习点**:
- 生产环境不应该有 mock 数据兜底
- 空状态 UI 比假数据更好
- AI 身份必须在 system prompt 中明确声明
- Python 脚本比手动编辑更安全（避免 edit_file 截断 bug）

## 关键原则总结

### 1. 数据真实性
- **开发环境**: 可以用 mock 数据方便调试
- **生产环境**: 必须禁用 mock，显示空状态或错误提示
- **用户信任**: 一旦用户发现数据是假的，信任就崩塌了

### 2. AI 身份一致性
- 每个 AI 必须有明确的身份声明
- IDENTITY.md 必须注入到 system prompt
- 避免 AI "人格混乱"（把自己当成别的 AI）

### 3. 代码修改安全
- 小改动：直接 edit_file
- 大改动：用 Python 脚本批量替换
- 修改前备份，修改后验证

### 4. 问题诊断流程
1. 收集症状（截图、日志、描述）
2. 定位代码（grep 搜索关键词）
3. 分析根因（为什么出现这个问题）

## 对话5：员工账号 404 问题 + DREAMING 模式
**链接**: https://www.coze.cn/s/FlooOPmKQ4w/

### 问题1：员工账号创建 AI 后 404
**症状**: 员工账号创建的 AI，刷新后显示 404

**根因**: 
1. 前端 localStorage 缓存了脏数据（不存在的员工 ID）
2. 员工账号创建的 AI 数据存在员工账号命名空间里
3. 后端 API 按用户隔离，老板账号看不到员工账号的数据

**解决方案**:
1. 清理浏览器 localStorage：`localStorage.clear()`
2. 刷新页面重新加载
3. 员工账号需要重新创建 AI

**关键发现**:
- 数据隔离：每个用户（token）有自己的数据空间
- 前端缓存：localStorage 可能缓存了旧数据
- 跨账号问题：老板和员工的数据不互通

### 问题2：DREAMING 模式
**症状**: 员工状态显示 "DREAMING"，但不知道什么意思

**根因**: DREAMING 是 UI 摆设，后端没有实现

## 对话6：Git 冲突 + Bug 修复 + AI 编造问题
**链接**: https://www.coze.cn/s/mPU_t2cvfKY/

### 问题1：Git 冲突
**症状**: `git pull` 失败，提示 divergent branches

**根因**: 之前 merge 留下了分歧，本地有未提交的改动

**解决方案**:
```bash
# 暂存本地改动
git stash

# 拉取最新代码
git pull origin dev --no-rebase

# 恢复本地改动
git stash pop
```

### 问题2：Bug 快速修复
**症状1**: `formatTimeAgo is not defined`
**修复**: 在 `renderMemoryTab` 前定义 `formatTimeAgo` 函数

**症状2**: `renderMemory is not defined`
**修复**: `switchEmpDetailTab` 里调用的是旧函数名，改成 `renderMemoryTab`

### 问题3：AI 编造问题（重要！）
**症状**: 员工详情面板显示黄色提醒 "⚠️ 该员工尚未配置身份文档..."

**根因**: 这是 AI **编造**的假任务数据！
- 系统没有 "任务数据" 这个概念
- 黄色提醒是 AI 自己编造的
- 用户被误导以为真的有问题

**Trumind 分析**:
```
这是 AI 编造的经典案例：
1. 系统没有 "任务数据" 这个概念
2. 黄色提醒是 AI 自己生成的假信息
3. 用户被误导以为真的有问题
```

**解决方案**: 在 System Prompt 里加禁止编造规则
```javascript
var soulContent = emp.soulDoc || emp.systemPrompt || 
  '你是 ' + emp.name + '，一个 ' + emp.role + '。' +
  '【铁律】禁止编造：你不知道的事情直接说不知道，不要猜测或编造信息。' +
  '请用第一人称回复，保持角色一致性。';
```

### 学习到的技巧

#### 1. Git 冲突解决流程
```bash
# 1. 暂存本地改动
git stash

# 2. 拉取最新代码
git pull origin dev --no-rebase

# 3. 恢复本地改动
git stash pop

# 4. 如果有冲突，手动解决后提交
git add .
git commit -m "merge: resolve conflicts"
```

#### 2. Bug 快速定位
- 控制台报错 → 找到具体行号
- 检查函数是否定义
- 检查函数名是否一致（renderMemory vs renderMemoryTab）
- 快速修复，不要过度工程化

#### 3. AI 编造识别与防范
**识别编造**:
- 信息来源不明
- 系统没有这个概念但 AI 提到了
- 过于具体的假数据

**防范编造**:
- 在 System Prompt 加禁止编造规则
- 要求 AI "不知道就说不知道"
- 定期检查 AI 回复的真实性

#### 4. System Prompt 规则注入
```javascript

  '请用第一人称回复，保持角色一致性。';
```
